结构化视图

所有结构化视图（格式）可以分为两种类型：

  • 层次结构（XML、JSON）- 单个文本文件；对象组的信息作为列表放在父组信息内部。
  • 平面（DBF、CSV、XLS、TABLE）- 每个对象组一个表文件，每个具有多个嵌套级别的对象组在其表中应有一个名为parent的列，其中应包含父组表中的"上一行"编号。
信息

处理层次深度大于1的平面格式不是很方便（因为在这种情况下，开发人员必须维护一个额外的列），因此，通常平面格式仅用于处理简单表单（层次深度小于1）。在其他情况下，通常使用层次结构格式。

格式还分为：

  • 人类可读（文本）和二进制。所有层次结构格式都是人类可读的，平面格式可以是二进制的（DBF、TABLE、XLS），也可以是人类可读的（CSV）。可以并且应该为人类可读格式指定编码（默认使用UTF-8）。
  • 标准化和内部格式。目前，仅支持一种内部格式：TABLE（值表）。所有其他格式都是标准化的。内部格式文件在某些集成操作中以特殊方式处理（例如，在SQL调用中）。此外，内部格式可用于lsFusion系统之间的通信。

在当前实现中，结构化视图中会忽略按列分组的平台。

在结构化视图中构建对象组层次结构时，忽略所有其所有对象在表单打开时传递的对象组（就好像这些对象组不存在一样）。

如果在结构化视图中显式指定了属性显示组，则该组不应早于默认组（如果指定的组出现得更早，则无论如何都会使用默认显示组）。

导出/导入名称​

表单上用于导出/导入的属性名称可以使用相应的选项（EXTID）显式指定。除非这样做，否则表单上的属性名称将用作导出/导入的名称。如果也未指定，则属性本身的名称将是导出/导入的名称（不像其他机制那样在其末尾添加参数对象 - 报表、交互式视图设计自定义等）。对象组和属性组的导出/导入名称的确定方式类似。

信息

由于语法性质，指定表单导入/导出名称的选项（特别是其空组对象）称为FORMEXTID（不是EXTID）。

信息

与表单上的属性名称不同，如果不同属性位于层次结构的不同节点（即不同的对象/属性组）中，则这些属性的导出/导入名称（EXTID）可以相等。这同样适用于对象组和属性组的导出/导入名称。

EXTID可以有专门的后缀：

  • ':escapeInnerJSON' 此后缀表示如果值包含JSON或类似JSON的内容（以方括号或花括号开头和结尾），我们将把它视为字符串而不是JSON。 示例： 
    regexpPattern '掩码' () = '[0-9]';
    FORM exportPattern
    PROPERTIES() c1 = regexpPattern EXTID 'c1', c2 = regexpPattern EXTID 'c2:escapeInnerJSON';
    {"c1":["0-9"],"c2":"[0-9]"}

层次结构视图​

在直接进行表单导出/导入之前，平台会构建属性、对象/属性组的层次结构，如下所示：

  • 对象/属性组的层次结构是根据对象组的层次结构和属性显示组构建的：属性显示组被视为此属性的父级，对象组的层次结构被保留。
  • 然后，对于每个X对象组：
    • 确定所有X后代所属的属性组，然后这些属性组及其祖先自动包含在层次结构中。此外：
      • 属性组成为属于这些组的X后代的父级
      • 属性组的层次结构被保留
      • X对象组成为使用的最上层（即没有父级）属性组的父级。
信息

在层次结构视图中，对象组可以像属性一样包含在属性组中。

信息

描述的算法与在默认设计中构建属性容器的算法非常相似（唯一的区别是它不使用对象组的层次结构）。与容器构建机制一样，同一属性组可以针对不同的对象组多次包含在层次结构中。

构建层次结构后，表单按照以下规则递归地导出/导入：

JSON:

JSON result ::= 
    { JSON with properties, groups of objects/properties without parents }

JSON with properties, groups of objects/properties ::= 
    JSON of the property 1 | JSON of the property group 1 | JSON of the object group 1
    JSON of the property 2 | JSON of the property group 2 | JSON of the object group 2
    ...
    JSON of the property M | JSON of the property group M | JSON of the object group M

JSON of the property ::=
    "property name on the form" : property value

JSON of the property group ::=
    "property group name" : { JSON with child properties, groups of properties/objects }

JSON of the object group ::=
    "object group name" : [
        { JSON with child properties, groups of properties/objects 1 }, 
        { JSON with child properties, groups of properties/objects 2 },
        ... 
        { JSON with child properties, groups of properties/objects N },
    ]

XML:

XML result ::= 
    <form name> XML with properties, groups of properties/objects without parents </form name>

XML with properties, groups of properties/objects ::= 
    XML of the property 1 | XML of the property group 1 | XML of the object group 1
    XML of the property 2 | XML of the property group 2 | XML of the object group 2
    ...
    XML of the property M | XML of the property group M | XML of the object group M

XML of the property ::= 
    <property name on the form> property value </property name on the form>

XML of the property group ::=
    <property group name> XML with child properties, groups of properties/objects </property group name>

XML of the object group ::=
    <object group name> XML with child properties, groups of properties/objects 1 </property group name>
    <object group name> XML with child properties, groups of properties/objects 2 </property group name>
    ...
    <object group name> XML with child properties, groups of properties/objects N </property group name>

在导出/导入XML时，可以为表单上的属性指定特殊的ATTR选项。因此，在导出/导入该属性时，其值不会存储在单独的标签中，而是存储在父标签的属性中：

<parent tag ... property name on the form = "property value" ...>

从XML导入时，最上层标签的名称（在规则中）被忽略（根据XML规范，应该只有一个这样的标签）。

具有NULL值的属性，以及作为导出结果在内部没有任何标签的属性组，不会被导出（被忽略）。

预定义值​

导入JSON时，如果对象组的值数组（[ ]）包含的不是对象（{ }），而是特定值（例如，数字或字符串），那么这个值会自动转换为对象{ "value" : value }。导出对象组到JSON时也会执行类似的转换：如果对象正好包含一个value键（即它的形式为{ "value" : value}），则在生成的JSON中会用这个value键的值代替它。除了"普通"对象组外，同样的转换也会对空根对象组执行，例如，JSON ["ab","vv"] 被处理为JSON { "value" : ["ab","vv"] }。

导入/导出XML时，如果属性名为value，则该属性的值将存储在父标签的内部（文本中），而不是在单独的标签中（即，就好像父标签本身就是属性视图一样）。如果父标签中有其他标签/属性（XML规范允许这样），通常会使用此行为。

导入XML时，如果对象组名为value，则会读取所有标签（无论名称如何）。

XML命名空间​

与其他格式不同，XML支持标签和属性的命名空间概念。

例如，在lsFusion中，要将属性导出到具有指定命名空间的标签，必须使用特殊语法指定此属性的名称：

[namespace[=uri]:]name

例如，h:table或h=http://www.w3.org/TR/html4:table。必要时命名空间名称可以为空。如果未为命名空间指定URI，则会从父标签中具有相同名称的命名空间继承。如果父标签中没有具有该名称的命名空间，则URI自动被视为等于http://www.w3.org/<命名空间名称>。

信息

无法在lsFusion语法中指定上述描述的属性名称（例如，h:table）（因为名称不能包含冒号），因此，要指定这样的导出名称，应使用上面描述的EXTID选项。

如果必须在标签中声明命名空间，但标签本身不应属于该命名空间，则必须向导出添加一个标记为ATTR且名为xmlns:namespace的属性。假设此属性的值将包含所声明命名空间的URI。

导入属性时，以及处理对象组/属性组时，使用命名空间的方式类似。

平面视图​

平面视图中每个对象组的文件是一个表，其中：

  • 行是此对象组的对象集合。
  • 列是属性，这些属性的显示组等于此对象组。

在CSV格式中（当没有第一标题行时），列的命名与XLS类似（即，A是第一列，B是第二列等）。

如果在导入表单时找不到具有表单属性名称的列，则选择表单属性列表中前一个属性列旁边的列进行导入（在这种情况下，parent列被视为第一个）。

语言​

所有上述选项，以及定义表单结构，都可以使用FORM语句完成。

打开表单​

要以结构化视图显示表单，使用相应的打开表单在结构化视图运算符。

示例​

FORM exportSku
    OBJECTS st = Store

    OBJECTS s = Sku
    PROPERTIES(s) id, name, weight
    FILTERS in(st, s)
;

exportSku (Store store)  {
    // 上传到DBF所有Sku，其中in (Store, Sku)为所需仓库指定
    EXPORT exportSku OBJECTS st = store DBF CHARSET 'CP866';
    EXPORT exportSku XML;
    EXPORT exportSku OBJECTS st = store CSV ',';
}

date = DATA DATE (INTEGER);
sku = DATA BPSTRING[50] (INTEGER);
price = DATA NUMERIC[14,2] (INTEGER);
order = DATA INTEGER (INTEGER);
FORM import
    OBJECTS o = INTEGER // 订单
    OBJECTS od = INTEGER // 订单行
    PROPERTIES (o) dateOrder = date // 从dateOrder字段导入日期
    PROPERTIES (od) sku = sku, price = price // 从sku和price字段导入产品数量
    FILTERS order(od) = o // 将顶部订单写入order

;

importForm()  {
    INPUT f = FILE DO {
        IMPORT import JSON FROM f;
        SHOW import; // 显示导入的内容

        // 在数据库中创建对象
        FOR DATE date = date(INTEGER io) NEW o = Order DO {
            date(o) <- date;
            FOR order(INTEGER iod) = io NEW od = OrderDetail DO {
                price(od) <- price(iod);
                sku(od) <- GROUP MAX Sku sku IF name(sku) = sku(iod); // 查找具有此名称的sku
            }
        }
    }
}
