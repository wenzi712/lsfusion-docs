值请求 (REQUEST)

值请求操作符创建一个动作，其中请求值与直接处理值是分开的。因此，如果在调用上下文中请求结果预先已知（System.requestPushed属性的值等于TRUE），则此操作符允许不执行值请求。这反过来又会在各种用例中自动为您提供以下功能：

  • 异步输入

    如果用户触发了属性更改事件（CHANGE），并且其处理过程恰好包含一个对内置类值的请求（最常见的情况），则平台不会立即调用它，而是先要求用户输入内置类的值。一旦完成此输入，用户可以立即继续其工作，并且输入结果会异步地（在新线程中）发送到服务器，在那里仅在此时（而不是在事件实际发生时）才开始处理发生的事件。此外（我们称这种技术为推送查询值），查询结果被标记为预先已知（System.requestPushed属性的值设置为TRUE），并且输入值被写入特殊属性族：requested。因此，由于此属性族用作所有值输入操作符的默认结果，因此模拟了同步处理；但是，表单上数据更改的人体工程学显著改善（例如，用户可以继续输入数据而无需等待表单上所有属性的计算）。

  • 粘贴（PASTE）

    当用户使用操作系统工具将值插入到属性单元格中时，平台会触发更改此属性的所见即所得事件（CHANGEWYS），并在服务器上调用相应的处理程序。在这种情况下，用户插入的值被推送为请求值。

  • 组合属性的更改事件（CHANGEWYS和CHANGE）默认处理程序

    如果正在更改的属性是使用带有一个参数的组合操作符创建的（通常是名称或ID），并且没有为其显式定义更改事件处理程序，则平台会自动按以下方式创建此处理程序：

    • CHANGE

      请求用户提供一个作为参数使用的属性值类的对象，并且该属性值更改为接收到的对象

    • CHANGEWYS

      请求用户提供一个已编辑属性值的类的对象，之后平台查找第一个组合属性值等于用户输入值的对象，然后将用作参数的属性的值更改为该对象。

    信息

    在这两种情况下，属性更改都意味着调用CHANGE事件处理程序，而属性更改到的值被推送为请求值。

    创建此类默认处理程序允许对值不仅属于内置类还属于自定义类的属性"开箱即用"地使用粘贴功能。

  • 组更改（GROUPCHANGE）默认处理程序

    默认情况下，组更改处理程序的创建方式如下：首先为对象的当前值调用CHANGE，然后如果输入未被取消，则对对象的所有其他值（与过滤器匹配）调用相同的处理程序，并将System.requestPushed属性的值设置为TRUE（假设输入的结果在处理期间不会更改，从而模拟了第一个值被推送的行为）。

信息

实际上，值请求操作符只执行两个操作：它为值请求操作检查System.requestPushed（推送值），为请求处理操作检查System.requestCanceled（取消值），并且还负责确定是否可以异步输入正在更改的属性。同时，使用此操作符使代码更清晰易读，因此建议使用它（而不是显式检查和选项）。

与其他值输入操作符一样，可以定义主操作和替代操作。第一个在输入成功完成时调用，第二个在未成功完成时（即输入被取消时）调用。因此，在此操作符中，正是这两个操作负责处理请求值。

隐式使用​

请注意，所有值输入操作符都可以通过使用相应的选项（DO）自动"包装"在值请求操作符中。而且，由于通常使用一个输入操作符输入值，因此建议默认使用此选项，仅在非常复杂的情况下才显式使用值请求操作符，例如，如果可能有多个输入选项（根据条件，不同的表单等）。

语言​

值请求操作符的语法由REQUEST操作符描述。

示例​


requestCustomer (Order o)  {
    LOCAL resultValue = STRING[100] ();
    REQUEST {
        ASK 'Choose from list?' DO
            DIALOG customers OBJECTS c = resultValue() CHANGE;
        ELSE
            INPUT = resultValue() CHANGE;
    } DO
        customer(o) <- resultValue();
}

FORM request
    OBJECTS o = Order
    PROPERTIES(o) customer ON CHANGE requestCustomer(o) // for example, group change will be performed
;
